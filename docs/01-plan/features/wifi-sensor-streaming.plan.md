# Plan: WiFi 센서 데이터 무선 스트리밍

> **기능명**: wifi-sensor-streaming
> **작성일**: 2026-05-29
> **상태**: Plan (Design 전)
> **대상**: 펌웨어 (ESP32-S3 + IIS3DWB)

---

## 1. 배경 / 목적

IIS3DWB 진동센서 디바이스를 **USB로는 전력만 공급**받고, **측정 데이터는 WiFi로
서버(라즈베리파이/PC)에 무선 전송**한다. 현재 펌웨어는 WiFi 연결까지만 되어 있고
(설정 툴로 WiFi 저장·연결 완료), 데이터 전송 기능은 없다.

```
[IIS3DWB] ─SPI→ [ESP32-S3] ─USB(전력만)
                     │
                     └─WiFi→ 📡 → [서버: 라즈베리파이/PC] (수집·저장·분석)
```

---

## 2. 목표 (요구사항)

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| FR-1 | 센서 가속도 데이터를 WiFi로 **라즈베리파이 서버**에 전송 | 필수 |
| FR-2 | 서버 IP/포트를 설정 가능 (NVS 저장 또는 설정 툴 연동) | 필수 |
| FR-3 | 연결 끊김 시 자동 재연결 | 필수 |
| FR-4 | **샘플레이트를 설정값으로 조절** (저속 ~ 26.6kHz 범위 선택) | 필수 |
| FR-5 | 전송 상태/처리량을 로그로 확인 | 중간 |
| NFR-1 | 데이터 유실 최소화 (또는 유실률 측정 가능) | 높음 |
| NFR-2 | 장시간(수 시간) 안정 동작 | 높음 |

### ⭐ 핵심 설계 방침: 샘플레이트는 "설정값"
26.6kHz를 고정하지 않고 **파라미터로 조절 가능**하게 한다. 같은 펌웨어로:
- **저속(예: 1kHz)** → 안정적, WiFi 부담 적음, 일반 모니터링용
- **고속(26.6kHz)** → 정밀 진동 분석용, WiFi 여건 좋을 때
- 사용자가 환경·용도에 맞게 선택. 단계적 검증과도 자연스럽게 맞물림.

> 서버: **라즈베리파이** (Python 수신 프로그램)

## 3. ⚠️ 기술적 제약 — 26.6kHz 원시 스트리밍의 현실

이 목표는 **대역폭/안정성 면에서 도전적**이다. 설계 전 반드시 검토한다.

### 대역폭 계산
```
26,667 샘플/초 × 3축 × 2바이트(int16) = 약 160 KB/초 = 1.28 Mbps  (순수 데이터)
+ 패킷 헤더/프로토콜 오버헤드 → 실제 1.5~2 Mbps 이상
```

### ESP32-S3 WiFi 현실
- 실측 TCP 처리량: 보통 1~10 Mbps (환경·혼잡도에 크게 의존)
- 1.28 Mbps는 **가능 범위이나, 끊김 없이 지속**하려면 정교한 설계 필요
- 센서 FIFO(512샘플, 3KB) 버퍼 활용이 필수
- WiFi 혼잡/약신호 시 데이터 유실 또는 버퍼 오버런 가능

### 결론
- 처음부터 26.6kHz 풀스트리밍을 노리면 디버깅이 매우 어렵다.
- **단계적 접근**으로 저속부터 검증 후 속도를 올린다 (아래 6장).

## 4. 전송 방식 후보 (Design에서 확정)

| 방식 | 장점 | 단점 | 26.6kHz 적합성 |
|------|------|------|----------------|
| **TCP 소켓** | 신뢰성(순서/재전송), 구현 표준적 | 오버헤드, 혼잡 시 지연 | ⚠️ 가능하나 튜닝 필요 |
| **UDP 소켓** | 낮은 오버헤드, 빠름 | 패킷 손실/순서 보장 없음 | ⚠️ 현실적, 손실 감수 |
| HTTP POST | 구현 쉬움, 서버 단순 | 오버헤드 큼 | ❌ 고속 부적합 |
| MQTT | IoT 표준, 대시보드 연동 | 브로커 필요, 고속 부적합 | ❌ |
| WebSocket | 양방향, 브라우저 친화 | 오버헤드 | ❌ 고속엔 부담 |

> **유력 후보**: TCP(신뢰성 우선) 또는 UDP(속도 우선). Design에서 실측 후 결정.

## 5. 데이터 포맷 후보 (Design에서 확정)

- **바이너리 패킷** (권장): `[헤더][샘플 N개 × 6바이트(x,y,z int16)]` — 효율적
- JSON: 사람이 읽기 쉽지만 26.6kHz엔 오버헤드 과다 → 고속엔 부적합
- 청크 단위: FIFO에서 N샘플 모아 한 패킷으로 전송 (오버헤드 감소)

---

## 6. 단계적 구현 계획 (핵심 전략)

처음부터 풀스피드를 노리지 않고, **검증 가능한 단계로 분할**한다.

### 1단계: 연결 + 저속 전송 (기반 검증)
- ESP32 → 서버 TCP/UDP 소켓 연결
- 저속(예: 100~1000 샘플/초)으로 가속도 데이터 전송
- 서버에서 수신·출력 확인
- **검증 포인트**: 무선으로 데이터가 실제로 도달하는가

### 2단계: FIFO 버퍼 + 청크 전송 (속도 향상)
- 센서 FIFO(512샘플) 활용 — 폴링 대신 묶음 읽기
- N샘플을 바이너리 청크로 묶어 전송 → 오버헤드 감소
- **검증 포인트**: 샘플레이트를 어디까지 올릴 수 있는가 (처리량 측정)

### 3단계: 고속 목표 도달 + 안정화
- 26.6kHz 목표 도달 시도
- 유실률 측정, 재연결, 버퍼 오버런 처리
- 장시간 안정성 테스트
- **검증 포인트**: 26.6kHz 지속 가능 여부 / 불가 시 현실적 최대치 확정

### 서버 측 (병행)
- 수신 프로그램 (Python: TCP/UDP 서버) — 라즈베리파이/PC
- 수신 데이터 파싱·저장(파일/DB)·간단 표시

## 7. 결정사항

### ✅ 확정됨
- **서버**: 라즈베리파이 (Python 수신 프로그램)
- **샘플레이트**: 고정 아닌 **설정값으로 조절** (저속~26.6kHz). 기본값은 안정적 저속으로 시작.

### ⬜ Design에서 답해야 함
- [ ] TCP vs UDP — 신뢰성 vs 속도 중 무엇 우선? (설정 샘플레이트에 따라 다를 수도)
- [ ] 서버 IP/포트 + 샘플레이트를 어떻게 설정? (NVS 저장 / 설정 툴 연동)
- [ ] 데이터 포맷 (바이너리 패킷 구조 정의)
- [ ] 샘플레이트 설정 범위/단계 (예: 1k / 5k / 13.3k / 26.6kHz)
- [ ] 센서를 폴링 방식 → FIFO/인터럽트 방식으로 바꿀지 (고속엔 FIFO 필수)
