1 기획 의도 (한 장 요약)

핵심: 진동센서를 USB로는 전력만 공급받고, 측정 데이터는 WiFi로 무선 전송하여 라즈베리파이 서버에서 수집한다. 설치 위치 제약(데이터 케이블 불필요) 없이, 전원만 있으면 무선으로 진동을 모니터링한다.
구분역할
USB 케이블전력 공급 (+ 최초 WiFi 설정 시 통신)
WiFi센서 데이터 무선 전송 (운영 중 주 채널)
라즈베리파이데이터 수신·저장·분석 서버

왜 무선인가? 진동 모니터링 대상(설비·기계)에 센서를 붙일 때 데이터 케이블을 길게 끄는 것은 비현실적이다. 전원(USB)만 공급하고 데이터는 무선으로 보내면 설치가 자유롭다.

2 데이터 흐름

┌──────────────── ESP32-S3 펌웨어 ────────────────┐ │ [IIS3DWB] ─SPI→ sensor_task ─(링버퍼)→ tx_task │ │ 진동센서 (생산자) (소비자) │ │ │ UDP │ │ 설정(NVS): 서버IP, 포트, 샘플레이트 │ │ └───────────────────────────────────────────────┼──────┘ USB(전력) ▼ WiFi [라즈베리파이] udp_receiver.py → 파싱 → CSV 저장 + 유실률 표시

생산자-소비자 분리가 핵심 설계다. 센서 읽기(SPI)와 WiFi 전송을 별도 태스크로 나눠, 한쪽이 느려도 다른 쪽을 막지 않는다. 중간 링버퍼가 속도 차를 흡수한다.

3 샘플레이트 5단계 (설정값)

26.6kHz로 고정하지 않고 설정값으로 조절한다. 같은 펌웨어로 환경·용도에 맞게 선택.

단계유효 레이트대역폭(3축×2B)방식 (실제 구현)용도
01 kHz~6 KB/s폴링일반 모니터링 (기본값)
13.3 kHz~20 KB/sFIFO + 1/8 데시메이션저속 분석
26.6 kHz~40 KB/sFIFO + 1/4 데시메이션중속
313.3 kHz~80 KB/sFIFO + 1/2 데시메이션고속
426.6 kHz~160 KB/sFIFO (전체)정밀 진동분석
왜 설정값인가? 26.6kHz는 WiFi 대역폭 부담이 커서 환경에 따라 불안정할 수 있다. 저속부터 선택 가능하게 하면 ① 안정 우선 환경은 저속, ② 정밀 분석은 고속으로 골라 쓰고, ③ 고속이 안 되는 환경에서도 낮춰서 계속 쓸 수 있다 (전부 못 쓰는 일 없음). 저장 위치: NVS (stream_rate, 0~4). 이 "설정값 조절" 기획 의도는 그대로 구현됨.
구현 메모: 센서 ODR/FIFO BDR이 26.6kHz 고정이라(데이터시트), 중간 단계(1~3)는 FIFO를 26.6kHz로 읽되 펌웨어가 N개당 1개만 전송(소프트웨어 데시메이션)하여 구현했다. 0단계만 1샘플 폴링, 1~4단계는 FIFO 버스트. 하나의 펌웨어가 stream_rate로 자동 분기.

4 패킷 포맷 (바이너리 UDP)

UDP 데이터그램 = 헤더(16B) + 샘플 배열. 효율과 손실 추적을 위해 바이너리 + 시퀀스 번호.

헤더 (16바이트)
  magic(4)="IIS3"  version(1)  rate_step(1)  sample_count(2)
  seq(4)           timestamp_ms(4)
페이로드 (sample_count × 6바이트)
  각 샘플: int16 x, int16 y, int16 z   (little-endian)

패킷당 200샘플 → 200×6 + 16 = 1216 바이트 (UDP MTU 안전)
설계 선택이유
원시 int16 전송 (mg 변환 X)디바이스 부담↓. mg 변환은 서버에서 (감도=풀스케일 의존)
seq(시퀀스 번호)UDP 패킷 손실/순서 추적 → 유실률 측정
바이너리 (JSON 아님)고속에서 JSON은 오버헤드 과다

7 기술 리스크 (정직한 검토)

26.6kHz 원시 스트리밍은 도전적이다. 160KB/s(=1.28Mbps) + 오버헤드를 ESP32-S3 WiFi로 끊김 없이 지속하는 것은 환경 의존적이다.
리스크대응
고속에서 WiFi가 못 따라감샘플레이트 5단계 → 낮춰서 사용 (전부 실패 방지)
UDP 패킷 손실설계상 감수. seq로 유실률 측정. 심하면 TCP 전환 또는 레이트 하향
폴링으로 고속 한계6.6kHz↑는 센서 FIFO 묶음 읽기로 전환 (예정)
26.6kHz 실측 미확정실측 전 보장 불가. 13.3kHz가 현실적 상한일 수 있음 → 단계 설정이라 무방
전략: 처음부터 풀스피드를 노리지 않고, 저속(1kHz)으로 "무선 도달" 먼저 검증 → 단계적으로 속도를 올리며 한계를 확인한다.

9 고속 샘플레이트 — 설정 메뉴 vs 실제 능력 (✅ 해결됨)

✅ 해결 완료 (2026-06-01): 아래 9.1~9.3은 당시 문제 상황과 의사결정 과정을 기록한 것입니다. 이후 FIFO 버스트 읽기 구현으로 26.6kHz까지 실기기 무손실 달성(드롭0). 메뉴-능력 불일치는 해소됐습니다. 상세 개선 내역: docs/html/high-speed-fifo-report.html.
[당시 문제] 설정 툴(GUI)에서 1k ~ 26.6kHz 5단계를 모두 선택할 수 있지만, 펌웨어가 폴링 방식이라 실제로는 저속(~1kHz)까지만 동작했음. (메뉴와 실제 능력 불일치 → 아래 과정으로 해결)

9.1 원인 — 폴링 방식의 구조적 한계

현재 센서 태스크는 iis3dwb_read_raw_data()로 1샘플씩 폴링하고 vTaskDelay()로 주기를 맞춥니다. FreeRTOS 틱이 CONFIG_FREERTOS_HZ=100 (1틱 = 10ms)이라 주기 제어 해상도가 거칠고, 고속에서 SPI 단발 읽기가 센서 출력을 못 따라갑니다.

설정 단계목표 레이트샘플 주기폴링으로 가능?
01 kHz1000 us실측 OK
13.3 kHz300 us불안정
26.6 kHz150 us사실상 불가
313.3 kHz75 us불가
426.6 kHz37 us불가

※ 센서 ODR은 26.6kHz 고정. 폴링이 못 쫓아가면 샘플을 놓쳐(언더샘플링) 실효 레이트가 떨어집니다.

9.2 채택한 해법 — FIFO 버스트 읽기 (✅ 구현 완료)

IIS3DWB는 내부 3KB FIFO(512샘플)에 26.6kHz로 자동 적재합니다. 이를 한 번의 SPI 버스트로 묶어 읽어 폴링 한계를 우회 → 고속 전송 달성.

  • ✅ 드라이버 FIFO 레지스터 정의 (FIFO_CTRL1~4, FIFO_STATUS, FIFO_DATA_OUT)
  • ✅ FIFO 읽기 함수 구현 완료 (iis3dwb_fifo_enable/count/read_fifo, 7B 워드 버스트)
  • ✅ sensor_streamer에 sensor_task_fifo 추가 (rate_step 분기)
  • ⚙️ 구현 중 발견·수정: CTRL1_XL XL_EN 비트 버그(100b→101b)가 FIFO 미적재 원인이었음

9.3 의사결정 옵션 (당시 검토 → A 선택)

옵션내용장점단점
A. FIFO 구현
(하이브리드, 권장)		 저속=폴링(현재 유지) + 중·고속=FIFO 버스트, rate_step으로 자동 전환 6.6k~26.6kHz 실제 달성, 메뉴가 진짜 동작, 검증된 1kHz 경로 보존 FIFO 읽기 구현 필요 (단, 기반 함수 일부 존재)

9.4 권장: 하이브리드 구조 (옵션 A의 구현 형태)

옵션 A는 폴링을 버리는 게 아니라, 폴링과 FIFO를 rate_step으로 자동 선택하는 하이브리드로 구현하는 것이 안전합니다. 사용자는 속도만 고르면 되고, 방식은 펌웨어가 알아서 전환합니다.

rate_step속도방식상태
01 kHz폴링 (현재 방식)검증됨, 그대로 유지
1~43.3k ~ 26.6kHzFIFO 버스트 읽기신규 구현
구조 — 생산자만 분기, 나머지 공통: 
[센서 태스크] ─→ [링버퍼] ─→ [전송 태스크] ─→ UDP ─→ 라즈베리파이
   ↑ 여기만 분기        ↑ 이하 전부 공통 (그대로 재사용)

 rate_step==0  →  sensor_task_polling()   (현재, 검증됨)
 rate_step>=1  →  sensor_task_fifo()      (FIFO 버스트, 신규)

하이브리드의 이점:

  • ✅ 검증된 1kHz 폴링 경로를 건드리지 않음 (회귀 리스크 0)
  • ✅ 링버퍼·전송·패킷 포맷 그대로 재사용 → FIFO 작업이 "센서 읽기"에만 집중
  • ✅ FIFO 미완성이어도 저속 제품은 즉시 출시 가능 (단계적 출시)
  • ✅ 기반 일부 존재: iis3dwb_read_registers()(연속 SPI), FIFO 레지스터 정의

신규 구현: iis3dwb_fifo_enable() + iis3dwb_read_fifo()(STATUS로 적재수 확인→DATA_OUT 버스트) + sensor_task_fifo() + start()의 rate_step 분기.

✅ 결과: 옵션 A(하이브리드) 선택·구현 완료. high-speed-fifo PDCA로 진행하여 26.6kHz 실기기 무손실 달성(드롭0/에러0). 저속 폴링 경로는 그대로 보존.
이후 fifo-interrupt PDCA로 하드웨어 인터럽트(INT1→IO4)도 실기기 검증 완료 — 효율 폴링과 진짜 인터럽트 둘 다 26.6kHz 무손실이며 사용자가 GUI에서 선택한다. 상세: docs/html/high-speed-fifo-report.html.

10 폴링 vs 인터럽트 — 둘 다 26.6kHz 검증, 사용자 선택

✅ 하드웨어 인터럽트 검증 완료 (2026-06-08): HW 담당자 확정 핀맵 INT1 → ESP32 IO4(INT2 → IO5 예비)에 watermark 인터럽트를 연결하여 26.6kHz 무손실 달성(샘플=267800/10s, 드롭0/에러0, INT 초당 ~409회). 이제 펌웨어는 효율 폴링과 진짜 하드웨어 인터럽트 두 경로를 모두 지원하며, 사용자가 GUI 설정 툴의 "데이터 읽기 방식"으로 선택한다(NVS read_mode).

10.1 두 가지 읽기 방식 (사용자 선택)

자동(폴링) read_mode=0Watermark 효율 폴링 (FIFO 버스트)26.6kHz 무손실기본값 · 배선 무관 안전
인터럽트·저부하 read_mode=1INT1(IO4) → ISR → 세마포어 (INT 올 때만 깨어남)26.6kHz 무손실CPU 여유 확보 · 멀티태스킹
fallback 안전장치: 인터럽트를 선택해도 INT 신호가 안 들어오면(배선 변경·오결선 등) 펌웨어가 자동으로 효율 폴링으로 강등(연속 미발생 감지)하여 스트리밍이 끊기지 않는다. 잘못 설정해도 데이터는 계속 흐른다.

10.2 구현 핵심 — watermark INT는 "레벨 신호"

검증 중 발견한 핵심: FIFO watermark 인터럽트는 레벨 신호(WTM 이상 채워진 동안 핀이 HIGH 유지)다. 따라서 엣지(POSEDGE) 트리거로는 최초 1회만 발동하고 멈춘다. 해결책:

  • GPIO_INTR_HIGH_LEVEL 레벨 트리거 사용
  • ISR은 INT를 비활성화하고 세마포어만 give (SPI 호출 금지, IRAM)
  • 태스크가 FIFO를 비운 뒤 INT를 재활성화 → 다음 채움에 다시 발동

이 패턴 전까지는 INT=1(한 번)에서 멈췄고, 적용 후 INT=4090/10s로 정상화됐다.

10.3 1kHz 같은 저속에서는 차이가 미미

인터럽트의 실질 이점은 고속/저부하다. 1kHz처럼 느리고 규칙적인 구간은 폴링·인터럽트 CPU 점유가 사실상 동일(~0.7%)하다. 그래서 기본값은 폴링으로 두고, 저부하가 필요한 사용자만 인터럽트를 선택하면 된다.

10.4 결론

정리: 효율 폴링·하드웨어 인터럽트 두 경로 모두 26.6kHz 무손실로 실기기 검증 완료. 사용자는 GUI에서 상황에 맞게 선택하고, 잘못 설정해도 fallback으로 안전하다. 현장 요구(저부하·고속·선택권)를 모두 충족한다. 상세: docs/02-design/features/fifo-interrupt.design.md.
