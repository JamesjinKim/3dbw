IIS3DWB 진동센서: USB는 전력만, 데이터는 WiFi로 라즈베리파이 서버에 전송

ESP32-S3 · ESP-IDF v5.4.3 · 기획 2026-05-29
목차 1. 기획 의도 (한 장 요약) 2. 데이터 흐름 3. 샘플레이트 5단계 (설정값) 4. 패킷 포맷 5. 펌웨어 구조 6. 라즈베리파이 수신 7. 기술 리스크 (정직한 검토) 8. 구현 현황 & 단계 9. ⚠️ 고속 설정 메뉴 vs 실제 능력 (의사결정 보류)

1 기획 의도 (한 장 요약)

핵심: 진동센서를 USB로는 전력만 공급받고, 측정 데이터는 WiFi로 무선 전송하여 라즈베리파이 서버에서 수집한다. 설치 위치 제약(데이터 케이블 불필요) 없이, 전원만 있으면 무선으로 진동을 모니터링한다.
구분역할
USB 케이블전력 공급 (+ 최초 WiFi 설정 시 통신)
WiFi센서 데이터 무선 전송 (운영 중 주 채널)
라즈베리파이데이터 수신·저장·분석 서버

왜 무선인가? 진동 모니터링 대상(설비·기계)에 센서를 붙일 때 데이터 케이블을 길게 끄는 것은 비현실적이다. 전원(USB)만 공급하고 데이터는 무선으로 보내면 설치가 자유롭다.

2 데이터 흐름

┌──────────────── ESP32-S3 펌웨어 ────────────────┐ │ [IIS3DWB] ─SPI→ sensor_task ─(링버퍼)→ tx_task │ │ 진동센서 (생산자) (소비자) │ │ │ UDP │ │ 설정(NVS): 서버IP, 포트, 샘플레이트 │ │ └────────────────────────────────────────────┼──────┘ USB(전력) ▼ WiFi [라즈베리파이] udp_receiver.py → 파싱 → CSV 저장 + 유실률 표시

생산자-소비자 분리가 핵심 설계다. 센서 읽기(SPI)와 WiFi 전송을 별도 태스크로 나눠, 한쪽이 느려도 다른 쪽을 막지 않는다. 중간 링버퍼가 속도 차를 흡수한다.

3 샘플레이트 5단계 (설정값)

26.6kHz로 고정하지 않고 설정값으로 조절한다. 같은 펌웨어로 환경·용도에 맞게 선택.

단계유효 레이트대역폭(3축×2B)방식용도
01 kHz~6 KB/s폴링일반 모니터링 (기본값)
13.3 kHz~20 KB/s폴링저속 분석
26.6 kHz~40 KB/sFIFO중속
313.3 kHz~80 KB/sFIFO고속
426.6 kHz~160 KB/sFIFO정밀 진동분석
왜 설정값인가? 26.6kHz는 WiFi 대역폭 부담이 커서 환경에 따라 불안정할 수 있다. 저속부터 선택 가능하게 하면 ① 안정 우선 환경은 저속, ② 정밀 분석은 고속으로 골라 쓰고, ③ 고속이 안 되는 환경에서도 낮춰서 계속 쓸 수 있다 (전부 못 쓰는 일 없음). 저장 위치: NVS (stream_rate, 0~4).

4 패킷 포맷 (바이너리 UDP)

UDP 데이터그램 = 헤더(16B) + 샘플 배열. 효율과 손실 추적을 위해 바이너리 + 시퀀스 번호.

헤더 (16바이트)
  magic(4)="IIS3"  version(1)  rate_step(1)  sample_count(2)
  seq(4)           timestamp_ms(4)
페이로드 (sample_count × 6바이트)
  각 샘플: int16 x, int16 y, int16 z   (little-endian)

패킷당 200샘플 → 200×6 + 16 = 1216 바이트 (UDP MTU 안전)
설계 선택이유
원시 int16 전송 (mg 변환 X)디바이스 부담↓. mg 변환은 서버에서 (감도=풀스케일 의존)
seq(시퀀스 번호)UDP 패킷 손실/순서 추적 → 유실률 측정
바이너리 (JSON 아님)고속에서 JSON은 오버헤드 과다

5 펌웨어 구조

컴포넌트역할상태
components/sensor_streamer/UDP 전송: sensor_task + tx_task + 링버퍼구현
components/config_manager/서버IP/포트/레이트 NVS 저장·로드 (확장)구현
main/main.cWiFi 연결+센서+서버설정 있으면 스트리밍 시작구현
전송 추상화 (UDP↔TCP)UDP 1순위, TCP는 함수 교체로 전환 가능UDP만
FIFO 묶음 읽기 (고속)6.6kHz↑용 iis3dwb_read_fifo()예정
오버런 처리: WiFi가 센서 속도를 못 따라가 링버퍼가 가득 차면, 가장 오래된 샘플을 버리고 드롭 카운트를 올린다 (시스템 멈춤 방지 + 유실 측정).

6 라즈베리파이 수신

rpi-receiver/udp_receiver.py — Python 표준 라이브러리만 사용 (추가 설치 불필요).

기능내용
수신UDP 포트(기본 9000)에서 패킷 수신, magic 검증
저장CSV: seq, timestamp_ms, sample_idx, x, y, z (raw 또는 mg)
모니터1초마다 실효 샘플레이트 / 유실률 실시간 표시
# 라즈베리파이에서
python3 udp_receiver.py                  # 포트 9000, raw CSV
python3 udp_receiver.py --mg --fs 4      # mg 단위 (±4g)

7 기술 리스크 (정직한 검토)

26.6kHz 원시 스트리밍은 도전적이다. 160KB/s(=1.28Mbps) + 오버헤드를 ESP32-S3 WiFi로 끊김 없이 지속하는 것은 환경 의존적이다.
리스크대응
고속에서 WiFi가 못 따라감샘플레이트 5단계 → 낮춰서 사용 (전부 실패 방지)
UDP 패킷 손실설계상 감수. seq로 유실률 측정. 심하면 TCP 전환 또는 레이트 하향
폴링으로 고속 한계6.6kHz↑는 센서 FIFO 묶음 읽기로 전환 (예정)
26.6kHz 실측 미확정실측 전 보장 불가. 13.3kHz가 현실적 상한일 수 있음 → 단계 설정이라 무방
전략: 처음부터 풀스피드를 노리지 않고, 저속(1kHz)으로 "무선 도달" 먼저 검증 → 단계적으로 속도를 올리며 한계를 확인한다.

9 ⚠️ 고속 샘플레이트 — 설정 메뉴 vs 실제 능력 불일치

현재 문제: 설정 툴(GUI)에서 1k ~ 26.6kHz 5단계를 모두 선택할 수 있지만, 펌웨어가 폴링 방식이라 실제로는 저속(~1kHz)까지만 정상 동작합니다. 3.3kHz 이상을 선택해도 그 속도가 나오지 않습니다. (메뉴와 실제 능력이 불일치)

9.1 원인 — 폴링 방식의 구조적 한계

현재 센서 태스크는 iis3dwb_read_raw_data()로 1샘플씩 폴링하고 vTaskDelay()로 주기를 맞춥니다. FreeRTOS 틱이 CONFIG_FREERTOS_HZ=100 (1틱 = 10ms)이라 주기 제어 해상도가 거칠고, 고속에서 SPI 단발 읽기가 센서 출력을 못 따라갑니다.

설정 단계목표 레이트샘플 주기폴링으로 가능?
01 kHz1000 us실측 OK
13.3 kHz300 us불안정
26.6 kHz150 us사실상 불가
313.3 kHz75 us불가
426.6 kHz37 us불가

※ 센서 ODR은 26.6kHz 고정. 폴링이 못 쫓아가면 샘플을 놓쳐(언더샘플링) 실효 레이트가 떨어집니다.

9.2 진짜 해법 — FIFO 버스트 읽기 (미구현)

IIS3DWB는 내부 3KB FIFO(512샘플)에 26.6kHz로 자동 적재합니다. 이를 한 번의 SPI 버스트로 묶어 읽으면 폴링 한계를 우회해 고속 전송이 가능합니다.

  • ✅ 드라이버에 FIFO 레지스터 정의는 이미 존재 (FIFO_CTRL1~4, FIFO_STATUS, FIFO_DATA_OUT)
  • ❌ 하지만 FIFO를 읽는 함수는 아직 구현 안 됨 (iis3dwb_read_fifo() 없음)
  • ❌ sensor_streamer도 폴링 기반 → FIFO 기반으로 교체 필요

9.3 의사결정 옵션 (보류 중 — 미선택)

옵션내용장점단점
A. FIFO 구현
(하이브리드, 권장)		 저속=폴링(현재 유지) + 중·고속=FIFO 버스트, rate_step으로 자동 전환 6.6k~26.6kHz 실제 달성, 메뉴가 진짜 동작, 검증된 1kHz 경로 보존 FIFO 읽기 구현 필요 (단, 기반 함수 일부 존재)
B. 메뉴 제한 GUI에서 3.3k 이상 숨김/비활성, 1kHz만 노출 메뉴-능력 불일치 즉시 해소, 안전 고속 기능은 당분간 미제공
C. 현상 유지
+ 경고		 5단계 유지하되 "3.3k 이상 실험적" 안내 변경 최소, 추후 결정 여지 사용자가 고속 선택 시 오동작 혼란 가능

9.4 권장: 하이브리드 구조 (옵션 A의 구현 형태)

옵션 A는 폴링을 버리는 게 아니라, 폴링과 FIFO를 rate_step으로 자동 선택하는 하이브리드로 구현하는 것이 안전합니다. 사용자는 속도만 고르면 되고, 방식은 펌웨어가 알아서 전환합니다.

rate_step속도방식상태
01 kHz폴링 (현재 방식)검증됨, 그대로 유지
1~43.3k ~ 26.6kHzFIFO 버스트 읽기신규 구현
구조 — 생산자만 분기, 나머지 공통: 
[센서 태스크] ─→ [링버퍼] ─→ [전송 태스크] ─→ UDP ─→ 라즈베리파이
   ↑ 여기만 분기        ↑ 이하 전부 공통 (그대로 재사용)

 rate_step==0  →  sensor_task_polling()   (현재, 검증됨)
 rate_step>=1  →  sensor_task_fifo()      (FIFO 버스트, 신규)

하이브리드의 이점:

  • ✅ 검증된 1kHz 폴링 경로를 건드리지 않음 (회귀 리스크 0)
  • ✅ 링버퍼·전송·패킷 포맷 그대로 재사용 → FIFO 작업이 "센서 읽기"에만 집중
  • ✅ FIFO 미완성이어도 저속 제품은 즉시 출시 가능 (단계적 출시)
  • ✅ 기반 일부 존재: iis3dwb_read_registers()(연속 SPI), FIFO 레지스터 정의

신규 구현: iis3dwb_fifo_enable() + iis3dwb_read_fifo()(STATUS로 적재수 확인→DATA_OUT 버스트) + sensor_task_fifo() + start()의 rate_step 분기.

현재 상태: 옵션 A(하이브리드)로 방향 확정 → 별도 PDCA(high-speed-fifo)로 구현 계획 진행. 1단계(1kHz)는 실기기 검증 완료, 저속 경로는 그대로 보존됩니다.
