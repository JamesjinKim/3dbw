진동센서의 전체 대역폭(26.6kHz)을 손실 없이 무선 전송할 수 있게 되었습니다. 이전에는 폴링 한계로 1kHz가 최대였습니다.

2. 이전 방식의 한계 (1샘플 폴링)

기존 sensor_task는 1샘플씩 읽고 vTaskDelay로 대기하는 폴링 방식이었습니다.

while (running) {
    iis3dwb_read_raw_data(&raw);   // 1샘플 읽기
    vTaskDelay(period);            // 주기 대기 ← 병목
}
문제원인결과
1kHz 이상 불가vTaskDelay가 틱(10ms) 단위 → 정밀 주기 제어 불가3.3k~26.6kHz 선택해도 미동작
샘플 누락SPI 단발 읽기가 센서 ODR(26.6kHz)을 못 따라감언더샘플링
메뉴-능력 불일치GUI는 5단계 노출하나 펌웨어가 못 냄사용자 혼란

3. 개선 3가지 — 숨은 병목을 차례로 해소

고속화 과정에서 3개의 독립적 병목을 순서대로 발견·해결했습니다.

① FIFO 버스트 읽기 도입

센서 내장 FIFO(512샘플)가 26.6kHz로 자동 적재 → 펌웨어는 쌓인 수십 개를 한 번의 SPI 버스트로 묶어 읽습니다. 타이밍은 센서가 책임지므로 폴링의 정밀도 한계가 사라집니다.

fifo_count(&avail);              // 쌓인 개수 확인
read_fifo(burst, avail, &got);   // 묶음으로 한 번에 읽기 (7B/샘플)

② FreeRTOS 틱 100Hz → 1000Hz

이전개선
CONFIG_FREERTOS_HZ100 (1틱=10ms)1000 (1틱=1ms)
FIFO 비우기 주기10ms (266샘플씩 — FIFO 오버런)1ms (27샘플씩 — 여유)

③ SPI 클럭 1MHz → 8MHz

처리량 계산: FIFO 워드 = 7바이트(TAG 1 + XYZ 6). 26.6kHz × 7B = 1.49 Mbps 필요.
SPI 1MHz로는 못 따라감(병목) → 8MHz로 상향하여 26.6kHz 완전 달성. (IIS3DWB 최대 10MHz)

4. 핵심 버그 발견 — 가속도계 enable 비트

가장 결정적인 수정. FIFO를 켜도 데이터가 전혀 안 쌓이던(avail=0) 근본 원인이었습니다.

데이터시트 Table 29: 가속도계 enable은 CTRL1_XL의 XL_EN[2:0] = 101b 여야 합니다. 그런데 기존 코드는 비트 7만(100b) 설정했습니다.

이전 (버그)수정
XL_EN 비트(1<<7) = 100b(0x05<<5) = 101b
출력 레지스터 읽기동작함 (그래서 못 발견)동작
FIFO 적재안 됨 (트리거 실패)정상

※ 100b 상태에서도 일반 출력 레지스터는 값을 내보내 1kHz 폴링은 멀쩡해 보였습니다. FIFO 배치는 정확한 101b에서만 트리거되어, 고속 구현 중에야 드러난 잠복 버그였습니다.

5. CPU 부하 — 현장 요구 반영

현장에서 "폴링보다 부하가 덜한 방식"을 요구했습니다. 두 가지 해법을 모두 구현·검증했고, 사용자가 GUI에서 선택합니다.

① Watermark 기반 효율 폴링 (기본값)
FIFO에 일정량 쌓일 시간만큼 vTaskDelay로 CPU를 양보(잠듦)하고, 깨어나 한 번에 묶음 읽기. → 빈 FIFO를 헛되이 확인하지 않아 부하가 크게 줄어듭니다. INT 배선과 무관하게 안전 동작.
② 하드웨어 인터럽트 (INT1 → IO4, 검증 완료)
HW 담당자 확정 핀맵 INT1 → ESP32 IO4에 watermark 인터럽트를 연결. FIFO가 임계치까지 차면 핀이 ISR을 깨워 그때만 묶음 읽기 → 가장 낮은 부하. 실기기 26.6kHz 무손실(샘플 267800/10s, 드롭0/에러0, INT 초당 ~409회) 달성.
방식INT 핀26.6kHzCPU 부하
1샘플 폴링 (이전)불필요❌나쁨
FIFO 효율 폴링 (기본값)불필요✅ 달성낮음
FIFO + 하드웨어 INT (IO4)IO4 (검증됨)✅ 달성가장 낮음

※ 핵심 구현 포인트: watermark INT는 레벨 신호라 GPIO_INTR_HIGH_LEVEL로 받고, ISR에서 INT를 끈 뒤 태스크가 FIFO를 비우고 다시 켜는 패턴을 사용합니다(엣지 트리거는 1회만 발동). INT 신호 미수신 시 자동으로 효율 폴링으로 fallback합니다. 상세: fifo-interrupt.design.md.

IIS3DWB는 ODR/BDR이 26.6kHz 고정(데이터시트 확인). 중간 단계는 펌웨어 소프트웨어 데시메이션 (N개당 1개 전송)으로 구현. 검증된 1kHz 폴링 경로는 건드리지 않아 회귀 위험 0.
